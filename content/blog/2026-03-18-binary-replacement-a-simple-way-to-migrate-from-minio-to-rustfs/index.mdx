> **Migration status update:** RustFS is in public beta. Do not treat it as an in-place, zero-downtime, or fully compatible binary replacement for MinIO. Back up first, test against a copy, pin exact versions, and keep a rehearsed rollback path for any production change.

RustFS is an Apache 2.0 S3-compatible object storage project built in Rust. Familiar S3 clients and some existing data workflows can make it a useful migration candidate, but S3 API similarity does not make storage formats, configuration, identity, events, or operational behavior interchangeable.

This guide is an evaluation path, not a cutover guarantee.

## Scope the test

Record the versions on both sides before changing anything. The example that originally accompanied this article used MinIO `RELEASE.2025-04-22T22-12-26Z`; the current RustFS website recommends a pinned public beta release from [GitHub Releases](https://github.com/rustfs/rustfs/releases). Re-run the full test for your actual MinIO build and exact RustFS beta tag.

Before starting:

1. Create and verify a restorable backup.
2. Clone or snapshot the data into an isolated environment.
3. Inventory buckets, versions, retention, IAM, lifecycle rules, replication, events, OIDC or LDAP, encryption, and application-specific S3 calls.
4. Define acceptance checks and a rollback deadline.
5. Stop writers or redirect them to a disposable test endpoint while validating.

## Start RustFS against a test copy

Download a specific beta asset rather than a mutable `latest` URL:

```bash
# Replace <beta-tag> and <asset> with an exact release and package from GitHub.
curl -LO https://github.com/rustfs/rustfs/releases/download/<beta-tag>/<asset>
```

After extracting the package, start RustFS with the copied data path and non-production credentials:

```bash
./rustfs /data/minio-copy \
  --address ":9000" \
  --console-enable \
  --console-address ":9001" \
  --access-key "<test-access-key>" \
  --secret-key "<test-secret-key>"
```

Keep the original MinIO data and service unchanged during this evaluation.

For a container test, pin the RustFS image to the same beta tag:

```yaml
services:
  rustfs-evaluation:
    image: rustfs/rustfs:<beta-tag>
    command: /data
    environment:
      RUSTFS_ADDRESS: 0.0.0.0:9000
      RUSTFS_CONSOLE_ENABLE: "true"
      RUSTFS_CONSOLE_ADDRESS: 0.0.0.0:9001
      RUSTFS_ACCESS_KEY: <test-access-key>
      RUSTFS_SECRET_KEY: <test-secret-key>
    ports:
      - "9000:9000"
      - "9001:9001"
    volumes:
      - ./minio-copy:/data
```

## What to verify

The internal data formats and control planes differ. Validate every required item rather than assuming automatic migration.

The migration work has covered evaluation paths for items such as bucket metadata, objects, tags, Object Lock, versioning, replication configuration, IAM, and lifecycle configuration. Coverage can change between beta releases, so verify each item with the pinned versions.

Do not assume automatic migration for:

- site replication;
- event notifications;
- MinIO-specific online configuration;
- LDAP and OIDC configuration;
- KMS keys or external security integrations;
- monitoring, alerting, upgrade, and disaster-recovery runbooks.

Also exercise the exact application operations shown in the [S3 compatibility overview](https://docs.rustfs.com/features/s3-compatibility/) and report gaps with a reproducible case.

## Plan a controlled cutover

A production plan should include:

- an application write-freeze or explicit replication strategy;
- integrity and object-count checks before and after cutover;
- client, identity, lifecycle, event, and encryption verification;
- monitoring for errors, latency, capacity, and recovery behavior;
- a documented rollback owner, trigger, and time window.

If any required behavior is unclear, keep the workload on the original system and open an issue in the [RustFS repository](https://github.com/rustfs/rustfs/issues) before proceeding.
