> **Current status:** RustFS is in public beta. Use this as a data-copy evaluation guide, not a zero-downtime migration guarantee. Pin exact versions, back up both configuration and data, validate required S3 operations, and rehearse rollback before a production cutover.

In the evolving landscape of object storage, **RustFS** is an Apache 2.0 alternative that teams can evaluate alongside MinIO. RustFS is built in Rust and exposes an evolving S3-compatible surface.

This guide outlines a technical process for copying data from an existing MinIO cluster to an isolated RustFS deployment before a controlled cutover.

## Why Migrate?

Before executing the migration, it is important to understand the technical drivers:

1.  **Licensing Compliance:** RustFS uses the Apache 2.0 license, allowing for broader integration in commercial and proprietary environments without the copyleft implications of MinIO’s AGPLv3.
2.  **Memory-Safe Implementation:** RustFS is written in Rust and gives operators a different implementation to benchmark against their own latency and throughput requirements.
3.  **S3 Evaluation Path:** Existing tooling such as SDKs, backup scripts, and `mc` can be tested against the operations each workload requires.

## Migration Tooling: The MinIO Client (`mc`)

The most robust tool for this migration is, ironically, the **MinIO Client (`mc`)**. It provides distinct advantages over generic S3 CLI tools:

*   **Resumability:** Automatically handles network interruptions.
*   **Data Integrity:** Includes checksum verification for transferred objects.
*   **Metadata Preservation:** Retains original object tags, content-types, and custom metadata.

## Prerequisites

*   **Source:** A running MinIO instance.
*   **Destination:** A running RustFS instance.
*   **Tooling:** `mc` installed on a host with network access to both clusters.
*   **Credentials:** Dedicated, time-limited migration credentials with only the permissions the copy and verification steps require.

### A Note on Access Credentials

Prefer dedicated migration credentials over normal root or admin credentials. Grant bucket listing, object read/write, metadata, and bucket creation only where the selected `mc` command requires them, then revoke the credentials after validation.

If a bucket exists on the source but not the destination, `mc` may require `s3:CreateBucket`. Add that permission to the temporary destination policy instead of broadening everyday operator credentials.

## Step-by-Step Migration

### 1\. Configure Remote Aliases

Define aliases for both storage clusters in the `mc` configuration.

```
# Configure the Source (MinIO)
mc alias set minio-old https://minio.example.com ADMIN_ACCESS_KEY ADMIN_SECRET_KEY

# Configure the Destination (RustFS)
mc alias set rustfs-new https://rustfs.example.com ADMIN_ACCESS_KEY ADMIN_SECRET_KEY
```

*Verification:* Run `mc ls rustfs-new` to confirm connectivity and permissions.

### 2\. Perform a Dry Run

Before initiating the transfer, simulate the operation to verify path resolution and permissions. The `--dry-run` flag lists the operations that would be performed without moving data.

```
mc mirror --dry-run minio-old/production-data rustfs-new/production-data
```

### 3\. Execute the Mirror

There are two primary strategies for the data transfer:

#### Option A: Static Transfer (Cold Storage)

For backup or archive data that is not actively being written to.

```
mc mirror minio-old/production-data rustfs-new/production-data
```

#### Option B: Active Synchronization Before a Write Freeze

For a rehearsal, the `--watch` flag can continue copying new objects after the initial sync. This does not make the cutover zero downtime: concurrent overwrites, deletes, versions, events, and configuration require separate verification, and a final write freeze is still needed for a controlled handoff.

```
mc mirror --watch --overwrite minio-old/production-data rustfs-new/production-data
```

*   `--watch`: Continuously replicates new objects.
*   `--overwrite`: Updates objects on the destination if they have changed on the source.
*   `--remove`: (Optional) Deletes objects on the destination if they are deleted from the source. Use with caution.

### 4\. Final Cutover

Once the data is synchronized:

1.  **Quiesce the Application:** Pause write operations to the application to ensure a consistent state.
2.  **Wait for Sync:** Allow `mc mirror` to process the final queue of pending objects.
3.  **Update Configuration:** Change the S3 Endpoint URL in your application settings to point to the RustFS instance (`https://rustfs.example.com`).
4.  **Restart & Verify:** Restart the application and verify data accessibility.

### 5\. Verification

After the cutover, verify data integrity using the `diff` command. This compares the source and destination metadata to identify any discrepancies.

```
mc diff minio-old/production-data rustfs-new/production-data
```

If the command returns no output, it found no differences in the fields it checks. Record separate object-count, version, metadata, retention, identity, event, and application-level validation results.

## Conclusion

RustFS offers a permissively licensed object storage option to evaluate. Use standard tools such as `mc` with scoped credentials, evidence-based acceptance checks, a write-freeze plan, and a tested rollback path.
